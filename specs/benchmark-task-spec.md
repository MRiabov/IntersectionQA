# Benchmark Task Specification: IntersectionQA

Note: This specification describes IntersectionQA v0.1. Shared geometry, split,
provenance, and verifier concepts may be reused by IntersectionEdit, but task
semantics here are QA-specific unless stated otherwise.

This document freezes the task definitions for the first IntersectionQA implementation. It is intentionally implementation-facing: prompt generation, answer parsing, metrics, and dataset schema should be derivable from this file without guessing.

Threshold constants, kernel tolerances, and exact numeric epsilon values belong in `specs/label_rules.md`. This file defines task semantics, required fields, and answer contracts.

## 1. Purpose

IntersectionQA is a benchmark for reasoning over executable CadQuery object code and assembly transforms. The benchmark asks whether a model can infer spatial relations such as interference, contact, clearance, containment, and overlap magnitude from code and rigid transforms.

Official labels come from precomputed geometry metadata produced offline by the dataset pipeline. Labels must not come from model execution during evaluation. Public closed-book prompts expose only code, transforms, task instructions, and allowed answer formats; labels, diagnostics, thresholds, and verifier outputs are stored separately in dataset fields.

## 2. Common Assumptions

These assumptions apply to every task unless a task explicitly overrides them.

- Units are millimetres.
- Each object is a rigid solid generated by CadQuery code.
- Rigid transforms use `translation`, `rotation_xyz_deg`, and `rotation_order`.
- `translation` is `[x, y, z]` in millimetres.
- `rotation_xyz_deg` is `[rx, ry, rz]` in degrees.
- `rotation_order` is `"XYZ"` for all v0.1 records.
- Euler rotations are XYZ order. Public prompts may state this in prose instead of
  repeating a `rotation_order` field, but stored metadata and executable scripts
  must agree with the prompt convention.
- Interference means positive-volume overlap.
- Touching at a face, edge, or point is not interference.
- Closed-book models must not execute code or call CAD/verifier tools.
- Public prompts expose object code and assembly transforms.
- Official answers, raw labels, diagnostics, and failure reasons are stored separately from public prompt text.
- Prompt generation must be a pure transformation of precomputed geometry metadata; it must not recompute CadQuery labels.

## 3. Shared Label Fields

All task families may use these raw geometry and diagnostic fields. Public task rows should include the fields needed for evaluation and slicing, but prompts must not reveal answer labels or diagnostics.

| Field | Type | Meaning |
| --- | --- | --- |
| `volume_a` | number or null | Volume of object A after validation, in cubic millimetres. Transform does not change this value. |
| `volume_b` | number or null | Volume of object B after validation, in cubic millimetres. Transform does not change this value. |
| `intersection_volume` | number or null | Positive volume of the exact intersection solid, in cubic millimetres. Zero means no positive-volume overlap under the active label policy. |
| `normalized_intersection` | number or null | `intersection_volume / min(volume_a, volume_b)`. Must be null if either volume is unavailable or non-positive. |
| `minimum_distance` | number or null | Minimum clearance distance between solids in millimetres. For intersecting solids this should normally be `0.0` or null if the distance query is not meaningful or unavailable. |
| `relation` | string | One of `disjoint`, `touching`, `near_miss`, `intersecting`, `contained`, `invalid`. |
| `contained` | boolean or null | True when one solid is fully inside the other, if containment detection is implemented. Null means not computed. |
| `contains_a_in_b` | boolean or null | True when all material of A is contained in B. False when checked and not contained. Null means containment detection was disabled. |
| `contains_b_in_a` | boolean or null | True when all material of B is contained in A. False when checked and not contained. Null means containment detection was disabled. |
| `aabb_overlap` | boolean or null | Whether transformed axis-aligned bounding boxes overlap. This is diagnostic only and is not an official intersection label. |
| `exact_overlap` | boolean or null | True iff `intersection_volume > epsilon_volume` under the record's `label_policy`. Diagnostic only; official labels still come from `relation` and task rules. |
| `boolean_status` | string | `ok`, `skipped_aabb_disjoint`, `failed`, or `not_run`. |
| `distance_status` | string | `ok`, `skipped_positive_overlap`, `failed`, or `not_run`. |
| `label_status` | string | `ok` when all labels required by the task are available; otherwise `invalid`. More specific failure categories belong in `failure_reason`. |
| `failure_reason` | string or null | Machine-readable failure reason. Null when `label_status` is `ok`. |

Task derivation must use `label_status`. Rows with unavailable required labels must either be excluded from that task family or assigned the task-specific invalid label where defined.

## 4. MVP Task Families

The MVP implementation targets exactly these task families:

- `binary_interference`
- `relation_classification`
- `volume_bucket`

All MVP tasks use CADEvolve-derived two-object assemblies and closed-book code-only prompts. Synthetic primitive assemblies are allowed only as golden fixtures, smoke/debug edge cases, or local fallback when CADEvolve is unavailable; they are not the main dataset source.

Compact examples in this document may use synthetic boxes for readability. Released benchmark rows should normally use CADEvolve provenance and source metadata.

### 4.1 `binary_interference`

**Task purpose**

Determine whether object A and object B have positive-volume interference after their assembly transforms are applied.

**Input shown to model**

- Common assumptions.
- Normalized CadQuery code for `object_a()` and `object_b()`.
- The transform for each object, using `translation` and `rotation_xyz_deg`.
- A small assembly helper or assembly function if that is the chosen prompt template.
- The binary question and answer format.

The prompt must not include `intersection_volume`, `normalized_intersection`, `minimum_distance`, `relation`, `aabb_overlap`, `exact_overlap`, `contained`, `contains_a_in_b`, `contains_b_in_a`, `label_status`, `failure_reason`, or numeric `label_policy` thresholds.

**Allowed answers**

- `yes`
- `no`

**Exact answer format**

The full parsed answer must be exactly one lower-case token:

```text
yes
```

or

```text
no
```

No prose, punctuation, JSON, XML tags, markdown, or explanation is valid.

**Label derivation**

- `yes` if `label_status == "ok"` and `relation` is `intersecting` or `contained`.
- `no` if `label_status == "ok"` and `relation` is `disjoint`, `touching`, or `near_miss`.
- Rows with `relation == "invalid"` or non-`ok` required label status must be excluded from this MVP task unless a later invalid-handling experiment explicitly includes them.

**Evaluation metric**

- Primary: accuracy over all evaluated rows, with invalid parsed outputs counted as wrong.
- Always also report invalid output rate.
- Report per-label accuracy by binary answer.
- Report per-difficulty and per-subset accuracy when fields are available.

**Edge cases**

- Face, edge, or point contact is `no`.
- Small positive clearance is `no`, including `near_miss`.
- Positive overlap above the volume epsilon is `yes`.
- Containment is `yes` when implemented because it implies positive-volume overlap.
- AABB overlap alone never implies `yes`.
- AABB non-overlap may be used by the generation pipeline as a safe diagnostic shortcut, but the task label still comes from stored label metadata.

**Example prompt skeleton**

````text
You are given two CadQuery object-construction functions and assembly transforms.

Assume:
- Units are millimetres.
- Euler rotations are XYZ order, in degrees.
- "interference" means positive-volume overlap.
- Touching at a face, edge, or point is not interference.
- Do not execute code.

Object code:
```python
{object_code}
```

Transforms:
object_a:
  translation: {translation_a}
  rotation_xyz_deg: {rotation_a}
object_b:
  translation: {translation_b}
  rotation_xyz_deg: {rotation_b}

Question: After the transforms are applied, do object_a and object_b have positive-volume interference?

Answer with exactly one token: yes or no
````

### 4.2 `relation_classification`

**Task purpose**

Classify the exact spatial relation between object A and object B after assembly transforms.

**Input shown to model**

- Common assumptions.
- Normalized CadQuery object code.
- Transform metadata or equivalent assembly code.
- Definitions of all relation labels.
- The relation question and answer format.

The prompt must not expose raw geometry values, diagnostics, official labels, containment flags, or numeric `label_policy` thresholds.

**Allowed answers**

- `disjoint`
- `touching`
- `near_miss`
- `intersecting`
- `contained`
- `invalid`

**Exact answer format**

The full parsed answer must be exactly one of the allowed lower-case labels. No extra text is valid.

**Label derivation**

- Use the stored `relation` field.
- `invalid` is allowed only when the dataset row intentionally includes invalid or unavailable geometry examples for this task.
- MVP exports should prefer `label_status == "ok"` rows and may exclude `invalid` rows from primary evaluation.

**Evaluation metric**

- Primary: multiclass accuracy over all evaluated rows, with invalid parsed outputs counted as wrong.
- Always also report invalid output rate.
- Report per-relation accuracy.
- Report per-difficulty and per-subset accuracy.

**Edge cases**

- `touching` has no positive-volume overlap, even if the objects share a face, edge, or point.
- `near_miss` has positive clearance below the configured near-miss threshold.
- `intersecting` has positive-volume overlap above epsilon.
- `contained` overrides ordinary `intersecting` if containment detection is implemented and the object is fully inside the other.
- `invalid` is not a model uncertainty label. It means the official geometry label is unavailable or failed.

**Example prompt skeleton**

````text
You are given two CadQuery object-construction functions and assembly transforms.

Assume:
- Units are millimetres.
- Euler rotations are XYZ order, in degrees.
- Interference means positive-volume overlap.
- Touching at a face, edge, or point is not interference.
- Do not execute code.

Choose the relation after transforms:
- disjoint: positive clearance above the near-miss threshold
- touching: no positive-volume overlap, but contact within tolerance
- near_miss: no contact or overlap, but small positive clearance
- intersecting: positive-volume overlap
- contained: one solid is fully inside the other
- invalid: geometry failed or the label is unavailable

Object code:
```python
{object_code}
```

Transforms:
{transforms}

Answer with exactly one label.
````

### 4.3 `volume_bucket`

**Task purpose**

Predict the bucket for normalized positive-volume overlap:

```text
intersection_volume / min(volume_a, volume_b)
```

This tests magnitude reasoning while avoiding raw-scale dependence.

**Input shown to model**

- Common assumptions.
- Normalized CadQuery object code.
- Transform metadata or equivalent assembly code.
- Definition of normalized intersection.
- The fixed bucket list and answer format.

The prompt must not expose volumes, intersection volume, normalized intersection, relation, containment flags, diagnostics, or numeric `label_policy` thresholds.

**Allowed answers**

- `0`
- `(0, 0.01]`
- `(0.01, 0.05]`
- `(0.05, 0.20]`
- `(0.20, 0.50]`
- `>0.50`

**Exact answer format**

The full parsed answer must be exactly one bucket string from the allowed list. Spaces inside bucket strings must match the canonical form above.

**Label derivation**

Use `intersection_volume`, `normalized_intersection`, and the active `label_policy`.
The policy-positive test is `intersection_volume > epsilon_volume`, where
`epsilon_volume = label_policy.epsilon_volume_ratio * min(volume_a, volume_b)`.

- `0` when `intersection_volume <= epsilon_volume`.
- `(0, 0.01]` when `intersection_volume > epsilon_volume` and `0 < normalized_intersection <= 0.01`.
- `(0.01, 0.05]` when `intersection_volume > epsilon_volume` and `0.01 < normalized_intersection <= 0.05`.
- `(0.05, 0.20]` when `intersection_volume > epsilon_volume` and `0.05 < normalized_intersection <= 0.20`.
- `(0.20, 0.50]` when `intersection_volume > epsilon_volume` and `0.20 < normalized_intersection <= 0.50`.
- `>0.50` when `intersection_volume > epsilon_volume` and `normalized_intersection > 0.50`.

Rows require `label_status == "ok"`, finite positive `volume_a` and `volume_b`,
non-null `intersection_volume`, and non-null `normalized_intersection`.

**Evaluation metric**

- Primary: bucket accuracy over all evaluated rows, with invalid parsed outputs counted as wrong.
- Always also report invalid output rate.
- Report per-bucket accuracy.
- Report per-difficulty and per-subset accuracy.

**Edge cases**

- `touching`, `near_miss`, and `disjoint` are bucket `0`.
- Contained examples usually map to `>0.50`, but the bucket must still be derived numerically.
- Values exactly on a boundary belong to the upper-inclusive bucket shown in the bucket name.
- Values below the volume epsilon are treated as `0` only through the official stored `normalized_intersection`.

**Example prompt skeleton**

````text
You are given two CadQuery object-construction functions and assembly transforms.

Assume:
- Units are millimetres.
- Euler rotations are XYZ order, in degrees.
- Normalized intersection means:
  intersection_volume / min(volume(object_a), volume(object_b))
- Touching without positive-volume overlap has normalized intersection 0.
- Do not execute code.

Object code:
```python
{object_code}
```

Transforms:
{transforms}

Question: Which bucket contains the normalized intersection volume?

Allowed answers:
0
(0, 0.01]
(0.01, 0.05]
(0.05, 0.20]
(0.20, 0.50]
>0.50

Answer with exactly one bucket string.
````

**Example task row fields**

```json

  "id": "intersectionqa_volume_bucket_000001",

  "geometry_ids": ["geom_000001"],
  "source": "synthetic",
```

## 5. Later Task Families

These task families are non-MVP. Their names, answers, and core semantics are reserved now so schema and prompt code can grow without renaming.

### 5.1 `clearance_bucket`

**Task purpose:** Predict the bucket for minimum positive clearance between two non-intersecting solids.

**Input shown to model:** Object code, transforms, definition of clearance, fixed clearance buckets from `specs/label_rules.md`, and exact answer format.

**Allowed answers:** Bucket labels defined in `specs/label_rules.md`; expected starter labels are `touching`, `<1mm`, `1-5mm`, `5-20mm`, `>20mm`, plus an explicit intersecting/invalid policy if included.

**Exact answer format:** Exactly one canonical clearance bucket string.

**Label derivation:** Use `minimum_distance` and `relation`. Intersecting rows should be excluded or mapped by a documented task policy. Touching is distinct from small positive clearance.

**Evaluation metric:** Bucket accuracy, invalid output rate, per-bucket accuracy, per-difficulty/subset accuracy.

**Edge cases:** Intersections, touching within epsilon, and unavailable distance queries must be handled by explicit row filtering or an explicit invalid/intersecting bucket.

**Example prompt skeleton:** Provide code/transforms, define clearance as the minimum distance after transforms, list buckets, ask for exactly one bucket.

**Example task row fields:** Same as MVP rows, with `task_type: "clearance_bucket"`, `answer`, `minimum_distance`, `relation`, `label_status`, and `failure_reason`.

### 5.2 `pairwise_interference`

**Task purpose:** For a multi-object assembly, return all object pairs with positive-volume interference.

**Input shown to model:** Code for all objects, object names, transforms, and pair naming convention.

**Allowed answers:** Normalized comma-separated pairs or `none`.

**Exact answer format:** `none`, or comma-separated normalized pairs such as `A-B,A-D,C-D`. Inside each pair, object names are sorted by the assembly's canonical object order. Pairs are sorted lexicographically/canonically. No spaces.

**Label derivation:** Include a pair if its stored pairwise `relation` is `intersecting` or `contained`.

**Evaluation metric:** Exact set match and pair-level F1. Also report invalid output rate and per-subset accuracy.

**Edge cases:** No collisions, multiple collisions, invalid pair labels, duplicate pairs, reversed pairs, and unknown object names.

**Example prompt skeleton:** Show multi-object assembly, define interference, ask for all interfering pairs using canonical names, answer `none` if there are no pairs.

**Example task row fields:** `geometry_ids` for all pair records or one multi-object geometry record, `object_names`, `pair_labels`, `answer`, `labels`, and `diagnostics`.

### 5.3 `ranking_normalized_intersection`

**Task purpose:** Rank several assemblies or variants from highest to lowest normalized intersection.

**Input shown to model:** 3-5 labelled options such as A, B, C, D; each option includes code/transforms but not labels.

**Allowed answers:** Compact letter string such as `DBCA`.

**Exact answer format:** A permutation of all option letters, with no separators and no repeated letters.

**Label derivation:** Sort options by stored `normalized_intersection` descending. Ties must be resolved by a deterministic policy in `specs/label_rules.md` or excluded from the task.

**Evaluation metric:** Pairwise ranking accuracy. Exact-order accuracy may be reported secondarily. Always report invalid output rate.

**Edge cases:** Ties or near-ties, zero-overlap options, contained examples, missing labels, repeated letters, omitted letters, and extra letters.

**Example prompt skeleton:** Show each option's code/transforms, define normalized intersection, ask for highest-to-lowest order, and require a compact letter string.

**Example task row fields:** `geometry_ids`, `option_ids`, `option_letters`, `answer`, per-option `normalized_intersection` in labels, and tie policy metadata.

### 5.4 `repair_direction`

**Task purpose:** Predict which single axis direction for a specified movable object would most directly remove or reduce interference.

**Input shown to model:** Two-object assembly, movable object name, fixed object name, and allowed directions.

**Allowed answers:** `+x`, `-x`, `+y`, `-y`, `+z`, `-z`.

**Exact answer format:** Exactly one allowed direction string.

**Label derivation:** The first implemented slice uses
`conservative_aabb_separating_translation_v01`. It emits rows only for
positive-overlap records (`intersecting` or `contained`) and computes the
single-axis translation needed to separate stored world-space AABBs for each
signed axis direction with `label_policy.epsilon_distance_mm` clearance. The
answer is the direction with the smallest absolute movement. Ties are resolved
deterministically in this order: `+x`, `-x`, `+y`, `-y`, `+z`, `-z`.
`no_valid_move` is reserved for a future policy with concrete verifier-backed
semantics and is not valid for this slice.

**Evaluation metric:** Accuracy over direction labels, invalid output rate, per-label accuracy, and per-difficulty/subset accuracy.

**Edge cases:** Multiple equally valid directions, containment, repairs
requiring rotation or diagonal movement, and cases where every axis direction
remains colliding under future exact policies. Already non-interfering
assemblies are excluded from the first slice.

**Example prompt skeleton:** Show code/transforms, state object B is movable, define the six directions, ask which direction should move B to remove positive-volume interference.

**Example task row fields:** `movable_object`, `fixed_object`, `answer`,
`repair_policy`, `candidate_direction_labels`, `selected_magnitude_mm`,
`selected_translation_vector_mm`, `candidate_moves`, and usual geometry
labels/diagnostics.

### 5.5 `tolerance_fit`

**Task purpose:** Decide whether an assembly satisfies a stated minimum clearance requirement.

**Input shown to model:** Two-object code/transforms, required clearance in millimetres, and pass/fail answer format.

**Allowed answers:** `yes` or `no`.

**Exact answer format:** Exactly `yes` or `no`.

**Label derivation:** `yes` only if `relation` is `disjoint` or `near_miss` as applicable and `minimum_distance >= required_clearance_mm`. Intersecting and touching examples are `no`.

**Evaluation metric:** Accuracy, invalid output rate, per-label accuracy, and per-clearance-threshold accuracy.

**Edge cases:** Exact threshold equality, touching, positive overlap, missing distance labels, and task rows where required clearance exceeds obvious assembly scale.

**Example prompt skeleton:** Show code/transforms, state required clearance, ask whether the assembly satisfies the clearance requirement, require `yes` or `no`.

**Example task row fields:** `required_clearance_mm`, `answer`, `minimum_distance`, `relation`, `label_status`, and usual provenance/diagnostics.

## 6. Answer Formats

Parsers must be strict and task-specific.

Common parser behavior:

- Strip leading and trailing ASCII whitespace from the model output.
- Do not extract answers from prose.
- Do not accept markdown, JSON, XML tags, quoted strings, explanations, punctuation, or multiple lines with extra text.
- Matching is case-sensitive.
- Any output that does not exactly match the task's canonical answer grammar is invalid.
- Invalid format is counted separately as invalid output and as incorrect for accuracy.

Canonical answer grammars:

| Format | Valid outputs |
| --- | --- |
| Binary answer | Exactly `yes` or `no`. |
| Relation answer | Exactly one of `disjoint`, `touching`, `near_miss`, `intersecting`, `contained`, `invalid`. |
| Volume bucket answer | Exactly one of `0`, `(0, 0.01]`, `(0.01, 0.05]`, `(0.05, 0.20]`, `(0.20, 0.50]`, `>0.50`. |
| Ranking answer | Compact uppercase letter string such as `DBCA`; must be a valid permutation of the presented option letters. |
| Pairwise answer | `none`, or normalized comma-separated pairs such as `A-B,A-C`; no spaces, no duplicates, no unknown names. |
| Repair answer | Exactly one of `+x`, `-x`, `+y`, `-y`, `+z`, `-z`, `none`. |

## 7. Relation Semantics

The exact thresholds for volume epsilon, distance epsilon, and near-miss clearance belong in `specs/label_rules.md`. The semantic meanings are fixed here:

- `invalid`: Geometry failed or the label is unavailable.
- `contained`: One solid is fully inside another, if containment detection is implemented.
- `intersecting`: Positive-volume overlap above epsilon.
- `touching`: No positive-volume overlap, but minimum distance is less than or equal to epsilon.
- `near_miss`: No contact and no positive-volume overlap, but positive clearance is below the near-miss threshold.
- `disjoint`: Positive clearance above the near-miss threshold.

Provisional relation logic:

```text
if geometry failed or required labels are unavailable:
    relation = invalid
elif containment detection is implemented and one solid is fully inside the other:
    relation = contained
elif intersection_volume > epsilon_volume:
    relation = intersecting
elif minimum_distance <= epsilon_distance:
    relation = touching
elif minimum_distance <= near_miss_threshold:
    relation = near_miss
else:
    relation = disjoint
```

If containment detection is not implemented, contained cases may be labelled `intersecting`; in that case `contained` should be null or false and the dataset metadata must state that containment detection was disabled.

## 8. Metrics

All evaluations must report the parser policy and task type.

Required metrics:

- Accuracy for `binary_interference`, `relation_classification`, `volume_bucket`, `clearance_bucket`, and `tolerance_fit`.
- Invalid output rate for every task.
- Per-label accuracy for classification and bucket tasks.
- Per-difficulty and per-subset accuracy when difficulty/subset fields are available.
- Pairwise ranking accuracy for `ranking_normalized_intersection`: compute the fraction of option pairs ordered correctly.
- Exact set match for `pairwise_interference`: the predicted pair set must exactly equal the gold pair set.
- Pair-level precision, recall, and F1 for `pairwise_interference`.

Accuracy denominator includes invalid model outputs unless a report explicitly states an alternate filtered metric. The primary benchmark score should treat invalid outputs as wrong.

## 9. MVP Scope Decision

Implementation should start with:

- `binary_interference`
- `relation_classification`
- `volume_bucket`
- CADEvolve-derived two-object assemblies
- Minimal synthetic fixtures only for golden tests, smoke/debug examples, and local fallback
- Closed-book code-only prompts

The MVP should not implement multi-object prompts, ranking prompts, repair-direction tasks, tolerance-fit tasks, rendering, or tool-assisted model execution as first-class benchmark tasks. Those are later extensions after the two-object geometry labels, prompts, parsers, metrics, and schema fields are stable.

# IntersectionQA Agent Notes

## What This Repository Is

IntersectionQA is Python tooling for building, validating, exporting, and evaluating a CAD spatial-reasoning benchmark. The v0.1 benchmark is code-only and two-object: models see CadQuery object-construction code plus rigid transforms, then answer questions about the exact relation between the transformed solids. Geometry labels are precomputed offline with CadQuery/OpenCASCADE and stored in dataset rows; evaluation must not recompute official labels from model outputs.

The codebase is also evolving into shared infrastructure for IntersectionEdit, a sibling family where the model predicts edits or repair actions that achieve target spatial states. IntersectionEdit is opt-in and draft/future-facing unless a config explicitly enables its task types.

## Highest-Authority Specs

Read `specs/` before changing behavior. If docs disagree, use this precedence:

1. `specs/label_rules.md` controls relation semantics, thresholds, precedence, bucket boundaries, failure reasons, and golden geometry cases.
2. `specs/schema.md` controls record fields, enum values, metadata, manifests, hashes, IDs, and validation requirements.
3. `specs/benchmark-task-spec.md` controls task-facing semantics, prompt contracts, answer formats, parser behavior, and MVP scope.
4. `specs/generation_policy.md` controls source policy, candidate generation, balancing, split grouping, diagnostics, and anti-patterns.
5. `specs/intersectionedit-task-spec.md` and `specs/intersectionedit-full-spec.md` describe edit/repair/action tasks; treat them as draft unless code/config has already implemented a slice.
6. `README.md`, `docs/`, `epics-and-stories*`, and planning documents are useful orientation but should be corrected to match the canonical specs above.

## Core Domain Rules

- Units are millimetres; volumes are cubic millimetres.
- v0.1 objects are rigid solids generated by CadQuery code.
- Transforms are `translation`, `rotation_xyz_deg`, and `rotation_order`; v0.1 rotation order is always `XYZ`.
- Interference means positive-volume material overlap. Touching at a face, edge, or point is not interference.
- Label precedence is: invalid, contained, intersecting, touching, near_miss, disjoint.
- Default policy: `epsilon_volume_ratio = 1e-6`, `epsilon_distance_mm = 1e-4`, `near_miss_threshold_mm = 1.0`.
- Binary interference answers derive only from `relation`: `contained` and `intersecting` are `yes`; `disjoint`, `touching`, and `near_miss` are `no`; `invalid` is excluded from normal MVP rows.
- Public rows must be self-contained for prompt, answer, labels, diagnostics, provenance, split metadata, and hashes. They must not require CadQuery, OpenCASCADE, caches, STEP/STL/mesh files, or render files to load official labels or evaluate predictions.
- Do not put STEP, STL, OBJ, glTF, B-Rep dumps, serialized CadQuery/OCC objects, meshes, or renders in default public JSONL rows. Debug artifacts are separate exports.

## MVP Task Scope

The frozen IntersectionQA v0.1 MVP tasks are:

- `binary_interference`
- `relation_classification`
- `volume_bucket`

The schema and implementation also include later/opt-in task types such as `clearance_bucket`, pairwise/ranking variants, repair tasks, target-clearance/contact/centroid moves, candidate edit tasks, and `tolerance_fit`. Keep v0.1 defaults narrow unless intentionally working on those opt-in paths.

## Package Map

- `intersectionqa/schema.py`: Pydantic models and row-level validation contracts. This is the executable schema backstop for answers, metadata, hashes, diagnostics, and implemented task slices.
- `intersectionqa/enums.py`: canonical enum strings for relations, statuses, task types, splits, failure reasons, and audit statuses.
- `intersectionqa/config.py`: deterministic dataset config loaded from YAML; config hashes intentionally exclude local paths/caches and other non-content settings.
- `intersectionqa/pipeline.py`: end-to-end composition for smoke/release dataset generation: load sources, validate objects, generate geometry records, materialize rows, balance, split, validate, export metadata/manifests/cards/parquet.
- `intersectionqa/sources/`: CADEvolve loading/normalization, synthetic fixtures, source validation, and validation caches.
- `intersectionqa/generation/`: assembly generation, CADEvolve candidate generation, and geometry label cache integration.
- `intersectionqa/geometry/`: CadQuery execution helpers, transforms, bounding boxes, and label computation.
- `intersectionqa/prompts/`: prompt materializers for QA, buckets, ranking, repair/edit, counterfactual, and fit tasks.
- `intersectionqa/splits/`: grouped split assignment and leakage audits.
- `intersectionqa/export/`: JSONL, metadata, schema, dataset card, balancing, and parquet writers.
- `intersectionqa/evaluation/`: parsers, metrics, AABB/OBB baselines, tool-assisted upper bounds, failure analysis, edit/repair verification, and GRPO/reward helpers.
- `intersectionqa/training/`: prompt features, reasoning-trace helpers, and sampling logic for SFT/GRPO data preparation.
- `scripts/`: CLI entry points and reports around the package modules, grouped by `dataset/`, `evaluation/`, `training/`, `experiments/`, `publishing/`, and `devops/`; shared script-only helpers live in group-local `internal/` packages. Prefer changing package code first, then keep scripts thin.
- `configs/`: reproducible smoke, repair smoke, validation, SFT, and GRPO configs.
- `.agents/skills/run-experiments/`: repo-specific skill for experiment
  execution, including manifests, local preflight, Vast.ai instance selection,
  remote bootstrap, SFT/GRPO launch, monitoring, artifacts, and teardown.
- `tests/`: pytest coverage for schema, labels, prompts, source loading/validation, generation, export, splits, metrics, scripts, rendering, rewards, and training helpers.

## Generation And Release Flow

Typical release-candidate flow:

1. Load CADEvolve source records from the extracted source directory or prepared subset.
2. Validate source objects in isolated workers with timeouts; CADEvolve programs are untrusted Python and must not be executed in the main generation process.
3. Generate candidate two-object assemblies and transforms deterministically from config and seed.
4. Compute exact geometry labels and diagnostics once per geometry record.
5. Materialize public task rows from geometry records without recomputing geometry.
6. Assign group-safe splits and run leakage audits.
7. Validate rows with Pydantic and export split JSONL, schema, metadata, manifests, dataset card, and parquet.
8. Build reports: stats, AABB/OBB baselines, tool-assisted upper bound, failure analysis, comparison tables, and edit verifier reports when edit rows exist.

Silent dropping is discouraged by spec. Failed source loading, validation, generation, labeling, and export attempts should be represented in manifests or reports with machine-readable failure reasons.

## Splits And Leakage

Rows carry enough grouping metadata to audit leakage: generator/family, object pair, assembly group, counterfactual group, edit counterfactual group, and related split group/family metadata for IntersectionEdit. Use grouped split helpers instead of ad hoc random splitting. When deriving inner train/eval splits for SFT or GRPO, use `intersectionqa.splits.grouped.partition_internal_train_eval_rows` or the existing preparation scripts so related variants stay inseparable.

## IntersectionEdit Notes

Implemented opt-in slices include conservative `repair_direction`/`repair_translation`, exact axis-aligned repair, vector/program repair formats, target-clearance repair/movement, target-contact movement, centroid-distance movement, and candidate edit selection/ranking. These rows rely heavily on structured metadata in `PublicTaskRow.metadata`; validators in `schema.py` enforce expected answer strings and policy metadata.

For repair release candidates, the exact verifier executes trusted dataset row scripts to reconstruct CadQuery geometry. Do not run verifier commands on untrusted JSONL rows without an external sandbox.

## Testing Guidance

- For schema, enum, prompt, parser, label-policy, split, and metrics changes, add or update focused pytest coverage.
- For generation/export changes, run targeted tests first, then a small smoke build if runtime and local CADEvolve sources permit.
- For answer-format changes, update validators, parsers, prompt text, tests, and relevant specs together.
- For threshold or label-precedence changes, treat it as dataset-version-sensitive and update `label_rules.md`, config/hash expectations, validation tests, and release reports.
- For CLI script behavior, keep package-level functions testable and cover script output/side effects in `tests/test_*_script.py` when practical.

## Local Cautions

- The repo can contain large archives and generated artifacts under `data/`; avoid broad commands over all of `data/` unless needed.
- Public dataset output lives under ignored `data/` paths. Do not accidentally commit generated release artifacts unless explicitly asked.
- The worktree may be dirty. Do not revert user changes.
- Keep generated row fields, prompt-visible transforms, and geometry-label transforms exactly aligned. If prompt values are rounded, those rounded values must be the values used for labeling, or the unrounded values must be exposed consistently.
- Prefer existing modules and scripts over inventing new parallel pipelines.
